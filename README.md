# KD AI Test Bench

MVP локального веб-стенда для короткого практического теста конкретных моделей PP-DocLayout-L, MinerU 2.5, Qwen3-VL-8B-Instruct и Qwen2.5-3B-Instruct на трёх комплектах конструкторской документации электронных плат. Цель — проверить эти заявленные модели, а не искать лучшие альтернативы.

## Возможности MVP

- загрузка PDF через простой HTML-интерфейс без React;
- извлечение имени файла, размера, числа страниц и базового числа изображений;
- сохранение `TestRun` в SQLite;
- отдельные адаптеры PP-DocLayout-L, MinerU 2.5, Qwen3-VL-8B-Instruct и Qwen2.5-3B-Instruct без реальных внешних вызовов;
- выбор режима запуска: `mock`, `remote_api`, `manual_import`, `local_cpu`; в `remote_api` пока возвращается понятная ошибка «Для модели не настроен endpoint или API-ключ»;
- mock-результаты нужны только для проверки MVP-пайплайна и не должны использоваться для оценки качества реальных моделей;
- запись реалистичных raw/normalized JSON в локальные data-каталоги;
- ручное заполнение input/output tokens, visual tokens, критических ошибок и итоговой оценки;
- экспорт всех запусков в CSV и просмотр JSON API.

## Запуск на Windows через PowerShell

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -e .[test]
copy .env.example .env
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

## Тесты

```powershell
pytest -q
```

Откройте http://127.0.0.1:8000 и загрузите PDF. Реальные PDF, результаты обработки и `.env` не коммитятся.

## Docker Compose

```powershell
docker compose up --build
```

Приложение будет доступно на http://127.0.0.1:8000.

## Структура

- `app/api` — маршруты загрузки, ручного обновления, JSON API и CSV-экспорта;
- `app/services` — адаптеры моделей, метрики, нормализация, валидация BOM;
- `app/models` — Pydantic-схемы;
- `data/input`, `data/raw_output`, `data/normalized`, `data/reports` — локальные рабочие каталоги, исключены из Git;
- `tests` — минимальные unit-тесты.

## Что требует ручной настройки

- реальные API-ключи и запуск MinerU/Qwen/Qwen3-VL не подключены намеренно;
- определение таблиц без OCR/парсеров не выполняется: `table_count` остаётся `null` / «не определено» до будущей интеграции;
- качество BOM проверяется простыми deterministic-правилами без LLM.


## Архитектура адаптеров и ограничения моделей

В проекте используются только четыре заявленные модели:

1. `PP-DocLayout-L`;
2. `MinerU 2.5`;
3. `Qwen3-VL-8B-Instruct`;
4. `Qwen2.5-3B-Instruct`.

Gemini, Groq и другие альтернативы не входят в планируемую архитектуру. Нельзя заменять заявленную модель другой моделью без отдельной строки сравнения и отдельного `TestRun`. Бесплатные API могут иметь лимиты, менять условия доступа или исчезнуть, поэтому `provider`, точный `model_id` и `model_revision` фиксируются в каждом `TestRun`.

Ожидаемые входы адаптеров:

- `PP-DocLayout-L` получает изображения страниц PDF и запускается отдельной кнопкой/выбором как независимый тест раскладки. Результат содержит JSON-блоки страницы: тип блока, координаты, номер страницы и `confidence`, если источник его вернул.
- `MinerU 2.5` получает исходный PDF.
- `Qwen3-VL-8B-Instruct` получает PNG/JPEG страницы или crop-фрагменты.
- `Qwen2.5-3B-Instruct` получает только очищенный структурированный JSON или текст после MinerU/Qwen3-VL. Она не должна самостоятельно извлекать BOM из raw HTML MinerU. Для неё доступны только задачи: статус комплекта, найденные документы, неподтверждённые документы, риски, вопросы клиенту и краткая рекомендация.
